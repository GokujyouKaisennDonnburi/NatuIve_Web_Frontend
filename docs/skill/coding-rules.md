# コーディング規約

## 概要

本ドキュメントでは、本プロジェクトで実装を行う際のコーディングルールを定義します。

実装時は、可読性・保守性・再利用性を重視し、既存コードとの一貫性を最優先としてください。

新しい実装パターンを導入する前に、必ず既存実装を確認してください。

---

## 基本方針

以下を優先順位として実装してください。

1. 既存実装との一貫性
2. 可読性
3. 保守性
4. 再利用性
5. パフォーマンス

「より良い書き方」よりも、「プロジェクト内で統一されている書き方」を優先してください。

---

## 命名規則

## コンポーネント

PascalCase を使用します。

```tsx
EventCard.tsx
UserProfile.tsx
SearchForm.tsx
```

---

## Hooks

`use` から始めます。

```ts
useEventList.ts
useAuth.ts
```

---

## Service

機能単位で管理します。

```ts
event.ts
user.ts
auth.ts
```

---

## 型

用途が分かる名前にします。

```ts
Event

EventRequest

EventResponse

EventFormValues
```

---

## 定数

共通定数は UPPER_SNAKE_CASE を使用します。

```ts
MAX_EVENT_TITLE_LENGTH

DEFAULT_PAGE_SIZE
```

---

## 変数・関数

camelCase を使用します。

```ts
eventList

selectedEvent

fetchEvents()
```

---

# TypeScript

## any は使用しない

any の使用は禁止します。

型が不明な場合は

- unknown
- Generics
- 型定義の追加

を検討してください。

---

## 型の再利用

既存の型がある場合は再利用してください。

同じ意味を持つ型を新しく作成しないでください。

---

## 型推論

TypeScript の型推論を活用してください。

不要な型指定は避けます。

---

# React

## Server Components

Server Components を基本とします。

`"use client"` は必要な場合のみ使用してください。

---

## コンポーネントの責務

1コンポーネント1責務を基本とします。

責務が増えた場合は分割を検討してください。

---

## Props

Props は必要最小限にしてください。

不要な Props の受け渡しは避けます。

---

## Hooks

Hooks の呼び出し順を変更しないでください。

React Hooks のルールを遵守してください。

---

# Atomic Design

コンポーネントは以下の責務を守ってください。

## Atoms

最小単位のUI

例

- Button
- Input
- Badge

---

## Molecules

Atoms の組み合わせ

---

## Organisms

画面の主要なUI

---

## Templates

画面構成

---

## Pages

Atomic Design における画面ルーティング層

---

# API

画面から直接 API を呼び出しません。

必ず Service を経由してください。

```
UI

↓

Hook

↓

Service

↓

API
```

---

# Hooks

以下を Hooks に切り出してください。

- API 呼び出し
- 状態管理
- 共通ロジック

UI のみで使用する状態は useState を利用してください。

---

# コンポーネント

## 再利用

新しいコンポーネントを作成する前に

- atoms
- molecules
- organisms

を確認してください。

再利用できる場合は新規作成しません。

---

## ファイルサイズ

1ファイルが極端に大きくなった場合は分割を検討してください。

明確な行数制限はありませんが、責務を基準に判断してください。

---

# Tailwind CSS

Tailwind CSS を使用します。

## 基本方針

- インライン style は使用しない
- 既存実装に合わせる
- 不要な class を追加しない

---

# MSW

API を追加・変更した場合は

- Handler
- Mock データ

を合わせて更新してください。

---

# エラーハンドリング

適切に例外処理を行ってください。

ユーザーへ表示するメッセージと、
開発者向けログは分けて考えてください。

---

# コメント

コードで「何をしているか」ではなく、
「なぜその実装なのか」を記述してください。

不要なコメントは追加しません。

---

# リファクタリング

リファクタリングでは

- 挙動を変更しない
- 可読性を向上させる
- 重複コードを削減する

ことを目的とします。

---

# コードレビュー

レビューでは以下を確認してください。

## 設計

- 責務は適切か
- ディレクトリ構成は適切か
- 再利用できるか

---

## 実装

- 命名
- 型安全
- 可読性
- 重複コード
- エラーハンドリング

---

## UI

- レスポンシブ対応
- アクセシビリティ
- Tailwind の統一

---

## パフォーマンス

- 不要な再レンダリング
- 不要な API 呼び出し
- メモ化が必要か

---

# AIへの指針

AIエージェントはコードを生成する際、以下を遵守してください。

- 既存コードの実装方針を優先する
- 命名規則を統一する
- 型安全を維持する
- 再利用可能なコンポーネントを優先する
- 必要以上のリファクタリングを行わない
- プロジェクトで採用されていないライブラリや実装方法を提案しない

不明な点がある場合は推測で実装せず、既存実装やドキュメントを確認してください。
